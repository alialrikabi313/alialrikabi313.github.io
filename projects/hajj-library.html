Hajj Libraryمكتبة الحجمكتبة الحج

قارئ نصوص يعمل دون إنترنت بالكامل، مع بحث دلالي بالذكاء الاصطناعي يعمل على الجهاز نفسه (نموذج BERT عبر ONNX، دون أي خادم)، فضلاً عن محرّك بحث صرفي بنيتُه من الصفر. منشور ومُصان على Google Play. Shipped and maintained on Google Play.

01نظرة عامةOverview

مكتبة الحج تطبيق قراءة علمية يعمل بالكامل دون اتصال. يأتي مزوّداً بقاعدة بيانات SQLite مضغوطة (~35MB) وقاموس لغوي (~9.5MB)، ويتيح قراءة آلاف الصفحات التراثية بأدوات قارئ متكاملة، بلا أي تتبّع أو إعلانات أو Firebase. Hajj Library is a scholarly reading app that works entirely offline. It ships a compressed SQLite database (~35MB) and a linguistic dictionary (~9.5MB), letting users read thousands of heritage pages with a full reader toolkit — with zero tracking, ads, or Firebase.

التحدّي الحقيقي ليس عرض النص، بل دقّة البحث: أن تتطابق الكلمات رغم اختلاف الهمزات والتاء المربوطة والألف المقصورة، وأن تتوسّع الجذور إلى مشتقّاتها، وكلّ ذلك بسرعة فورية على الهاتف ودون خادم. ولهذا بنيتُ خطّ معالجة صرفي من أربع مراحل، وأضفتُ في الإصدار 2.1 بحثاً دلالياً يدمج التشابه المتّجهي مع البحث النصّي. The real challenge isn't rendering text — it's searching correctly: matching words across hamza, tā' marbūṭa, and alef-maqsūra variants, expanding roots into their derivatives, all instantly on-device with no server. I built a four-stage morphological pipeline for this, and in v2.1 added semantic search that fuses vector similarity with lexical retrieval.

03المعماريةArchitecture

feature-first + طبقات نظيفة، مستودعات ثابتة (static) لتبسيط الـ isolates والاختبارFeature-first + clean layers, static repositories for isolate-safety and testing

// 14 ميزة مستقلة · feature-first
lib/
├─ main.dart            init: settings → hydrate derivatives cache → maintenance
├─ core/               theme · arabic morphology utils · shared widgets
└─ features/
   ├─ books/           reader engine · TOC · page cache (10-page cap)
   ├─ search/          FTS5 + morphology orchestrator (16 files)
   │   └─ two-phase: fast (no derivatives) → if <50 hits → full morphology
   ├─ semantic_search/ ONNX embedder · vector index · rank fusion (v2.1)
   ├─ highlights · notes · bookmarks · collections · clippings
   ├─ reading_stats · reading_books · drawings
   └─ backup/          JSON export/import · AES · transactional rollback

Search flow:  input → SearchUtils → MatchQueryBuilder → SearchDb
            → 1000+ FTS rows → background isolate (score + snippet)
  • كشف المخطّط بثلاث طبقات: يتكيّف تلقائياً مع ثلاث بنى لقاعدة البيانات (جديدة بجداول pages + pages_fts، قديمة بـ FTS5، واحتياطية SQLite عادية) دون أن يتعطّل، مع ترقية الإصدار عبر ثابت _dbVersion.Three-tier schema detection: auto-adapts to three database layouts (new pages + pages_fts, legacy FTS5, plain-SQLite fallback) without breaking, with version migration via a _dbVersion constant.
  • مستودعات ثابتة بلا حاوية DI: الحالة في حقول static + SharedPreferences، ما يبسّط الاختبار والتوافق مع الـ isolates.Static repositories, no DI container: state in static fields + SharedPreferences, simplifying tests and isolate compatibility.
  • بحث على مرحلتين: المرحلة الأولى سريعة وبلا توسعة جذور، فإن أعادت ≥50 نتيجة توقّفت، وإلا فعّلنا المرحلة الثانية بالصرف الكامل.Two-phase search: phase one runs without root expansion (fast); if it returns ≥50 results it stops, otherwise full morphology kicks in.

04أبرز الميزات التقنيةKey technical features

01

خطّ صرف من الصفرFrom-scratch morphology engine

تطبيع للهمزات، ومطابقة للحروف المتشابهة (ة/ه، ى/ي، ض/ظ)، وتوسيع للجذور إلى مشتقّاتها عبر قاموس مدمج بحجم 9.5MB، مع ذاكرة LRU وكاش دائم يجنّبان إعادة تحليل الكلمات.Hamza normalization, confusable matching (ة/ه, ى/ي, ض/ظ), and root→derivative expansion via an embedded 9.5MB dictionary — with LRU and persistent caching to avoid re-analyzing words.

02

🧠 بحث دلالي بالذكاء الاصطناعي على الجهاز🧠 On-device AI semantic search

نموذج embeddings (BERT) يعمل محلياً على الجهاز نفسه عبر ONNX Runtime (مكمَّم int8)، دون أي خادم أو API سحابي. يدمج التشابه المتّجهي مع صلة FTS5 (rank fusion)، ويتراجع بسلاسة إلى FTS عند غياب النموذج. وتشغيل ML على الهاتف ذاته أصعب وأندر من مجرّد استدعاء API.A BERT embeddings model running locally on the device via ONNX Runtime (int8-quantized) — no server, no cloud API. It fuses vector similarity with FTS5 relevance (rank fusion), with graceful fallback to FTS. Running ML on the phone itself is harder and rarer than just calling an API.

03

offline-first بالكاملFully offline-first

قاعدة SQLite مضغوطة بحجم 35MB تُستخرَج وقت التشغيل، ووضع WAL، وتخزين مؤقّت للصفحات بسقف ذاكرة، وحفظ تلقائي لموضع القراءة، دون أي اعتماد على الشبكة.A 35MB compressed SQLite DB extracted at runtime, WAL mode, page caching with a memory cap, auto-saved reading position — with zero network dependency.

04

حماية بيانات حقيقيةReal data protection

نسخ احتياطي بمعاملات قابلة للتراجع (rollback)، كتابة متسلسلة آمنة لـ SharedPreferences، قراءة JSON آمنة (المدخلات التالفة لا تُسقط التطبيق)، وتشفير AES اختياري.Transactional backup with rollback, serialized safe SharedPreferences writes, safe JSON deserialization (corrupt entries don't crash the app), and optional AES encryption.

05

أدوات قارئ غنيةRich reader toolkit

تظليل بأربعة ألوان، ملاحظات بوسوم وتصدير Markdown، إشارات ملوّنة، طبقة رسم/تأشير، تمرير متّصل، نوافذ حواشٍ، 4 خطوط، وإحصاءات قراءة بأهداف يومية.Four-color highlights, tagged notes with Markdown export, colored bookmarks, a drawing layer, continuous scroll, footnote popups, 4 fonts, and reading stats with daily goals.

06

جاهزية إنتاجProduction readiness

إعداد توقيع APK، وسياسة خصوصية منشورة مطابِقة لمعرّف الحزمة، وتكامل مستمر عبر GitHub Actions (تحليل + اختبار + بناء)، وتثبيت متعمّد لإصدارات SQLite يتفادى فشل البناء على الشبكات البطيئة.APK signing config, a published privacy policy matching the package ID, GitHub Actions CI (analyze + test + build), and deliberate SQLite version pinning to avoid build failures on slow networks.

05قرارات هندسيةEngineering decisions

لماذا مستودعات static بدل Riverpod/Bloc؟Why static repositories instead of Riverpod/Bloc?
التطبيق أحادي المستخدم ويعمل دون اتصال، ومن ثمّ فإنّ إطار إدارة حالة ثقيلاً سيكون هندسة زائدة. الحقول الثابتة + SharedPreferences تُبقي المنطق قابلاً للاختبار ومتوافقاً مع الـ isolates التي تعالج آلاف صفوف البحث في الخلفية. وهذا القرار موثّق في ARCHITECTURE.md. The app is single-user and offline; a heavy state framework would be over-engineering. Static fields + SharedPreferences keep logic testable and compatible with the isolates that process thousands of search rows in the background. The decision is documented in ARCHITECTURE.md.
تثبيت إصدارات SQLite عمداًDeliberate SQLite version pinning
إصدارات sqlite3_flutter_libs الأحدث تستخدم native-assets hooks تفشل على الشبكات المتقطّعة وقت البناء، لذا ثبّتُّها على 0.5.x عمداً. تفصيل صغير، لكنه يعكس فهماً لسلسلة البناء وظروف العمل المحلية. Newer sqlite3_flutter_libs versions use native-assets hooks that fail on intermittent networks at build time, so it's pinned to 0.5.x on purpose. A small detail, but it reflects an understanding of the build chain and local working conditions.
